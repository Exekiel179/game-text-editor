# 心理健康训练游戏（第一章）详细开发文档（GDD v1）

## 0. 文档目标
将《基础游戏介绍》转化为可执行的开发规范，覆盖：3C基底、关卡与命名、对话系统与AI交互、训练Boss战、音视频策略、技术架构（Godot 4 + 胶水开发）、测试与验收标准。

---

## 1. 项目概述（Vision）
- **项目定位**：2D像素风心理健康训练游戏。
- **核心主题**：家庭冲突情境下的认知行为训练（CBT）。
- **第一章目标**：让玩家在可交互对话中掌握“事实 vs 想法”的基础认知识别能力。
- **叙事钩子**：玩家拥有“仅自己可见”的小宠物精灵，作为引导者、反馈者与情绪缓冲角色。
- **引擎与平台**：Godot 4（PC优先）。

### 1.1 一句话Pitch
玩家在家庭冲突现场中，通过与NPC对话和精灵引导，识别扭曲认知、区分事实与想法，并完成一场“认知训练Boss战”。

---

## 2. 第一章核心体验与30秒循环（Core Loop）
1. **行动（Action）**：探索场景 → 触发对话节点 → 选择回应/输入AI文本。
2. **反馈（Feedback）**：精灵即时标注“事实/想法/情绪词/推断跳跃”。
3. **奖励（Reward）**：认知清晰度值上升、冲突降级、剧情推进、解锁新对话策略。
4. **重复（Repeat）**：进入下一轮冲突片段，提升识别与表达能力。

---

## 3. 3C基底锁定（Character / Camera / Control）

## 3.1 当前阶段诊断
- 当前处于**前期可修改阶段（Pre-production）**，允许迭代3C。
- 一旦进入第一章内容量产（Production），3C应冻结，避免关卡与UI返工。

## 3.2 功能-形式映射表
| 功能需求 | 形式建议（角色/视觉） |
|---|---|
| 对话训练为核心 | 主角像素形体简洁，强调“可读性”而非战斗夸张动作 |
| 心理支持引导 | 精灵宠物采用高对比色和柔和动画，始终可被玩家快速识别 |
| 冲突情境观察 | 角色待机与转向动作细腻，突出“倾听/思考”状态 |
| 认知纠偏反馈 | 精灵可触发浮层标记（事实/想法标签气泡） |

## 3.3 Control（操控）
- **移动**：上下左右（键盘/手柄方向）。
- **交互**：
  - `Interact`：发起对话/确认选择。
  - `Journal`：打开“认知记录本”（事实/想法卡片）。
  - `Pet Hint`：请求精灵提示（有限次数或冷却）。
- **对话输入**：
  - 预设选项 + 可选自由输入（AI解析后映射到意图标签）。

## 3.4 Camera（镜头）
- 2D跟随镜头，轻微平滑，避免晕动。
- 对话触发时镜头短暂“注意力聚焦”（人物中景+UI高亮）。
- 冲突升级时镜头不抖动过强，优先可读性（心理主题避免刺激性过载）。

## 3.5 Character交互循环
- 探索 → 发现冲突线索 → 对话交互 → 精灵标注认知类型 → 玩家修正表达 → 冲突张力变化。

## 3.6 失败与反馈机制
- **失败不惩罚挫败**，采用“训练式回路”：
  - 错误分类（把想法当事实）→ 给出解释提示 + 示例重构句。
  - 连续错误 → 精灵进入“引导模式”（更明确脚手架提示）。
- **硬失败条件**（关卡层）：在关键节点多次未完成识别目标，进入“复盘段落”后重试。

## 4. 叙事结构与可扩展性评估（分支树可行性）

## 4.1 结论
第一章**不建议使用纯分支树**作为长期架构；可在局部短链路使用。

## 4.2 原因
- 你需要维护对话库 + 支持AI自由输入，纯分支树会指数膨胀。
- 以平均每节点3选项、深度8估算：`3^8 = 6561` 节点，维护成本过高。

## 4.3 推荐结构（可扩展）
采用**“状态机 + 标签变量收束 + 模板对话”混合模型**：
1. **剧情状态机（有限状态）**：开场冲突 / 升级 / 冷却 / 修复。
2. **认知变量层**：事实识别率、情绪强度、沟通安全度、信任值。
3. **意图标签路由**：AI输入先做意图识别（澄清、指责、回避、求证等）再映射到状态转移。
4. **收束策略**：多个表达进入同一“训练目标节点”，避免内容爆炸。

## 6. 训练Boss战规范（第一章结尾）

> 这里的Boss不是怪物，而是“高压冲突情境本身”。

## 6.1 【机制与反馈】
- **击败条件**：
  1. 在关键对话中连续完成指定数量的“事实/想法正确区分”；
  2. 使用一次“重构表达”（把指责改为观察+感受+需求）；
  3. 将冲突强度从高压降到可对话区间。
- **伤害表现**：
  - 错误识别会提升“冲突压力条”；
  - 正确识别降低压力条并提升“清晰度条”。

## 6.2 【资产需求表】
- **特效库**：
  - 认知命中提示（绿色）
  - 误判警示（橙/红）
  - 压力升降条动画
- **音频库**：
  - 正确反馈音（轻快短提示）
  - 误判反馈音（不刺耳）
  - 阶段切换音（冲突升级/缓和）
- **独立对象**：
  - 精灵引导特效
  - 关键词高亮框
- **HUD增量**：
  - 冲突压力条
  - 认知清晰度条
  - 当前训练目标提示

## 6.3 【玩家侧适配】
- 新增“暂停-复盘”入口（非惩罚）。
- 新增“精灵求助”按钮与冷却显示。

## 6.4 【场地与结算】
- 场地：家庭餐桌/客厅双区切换。
- BGM：平稳背景 + 冲突峰值层 + 缓和收束层。
- 奖励：解锁“认知记录本第1页”、新表达模板、下一章预告。

## 7. 音乐与音效来源策略（预算敏感型）

## 7.1 推荐路线
**混合方案（商用库 + 少量定制）**
- BGM主体：商用免版税库（低成本、高效率）。
- 关键主题（精灵引导/章节结尾）：小规模定制（外包或合作）。
- UI与反馈SFX：商用库为主，统一后期处理保证风格一致。

## 7.2 执行路径
1. 先锁定第一章音频需求表（按场景和UI事件）。
2. 从商用库挑选“家庭冲突低刺激、情绪分层明确”的曲包。
3. 仅对“精灵主题旋律”定制，建立IP识别度。
4. 统一混音标准（音量、动态、频段）。

## 7.3 风险提示
- 后期才补音频会导致训练反馈“手感断裂”。
- 使用知名版权曲成本高、授权周期长，不适合当前阶段。

## 8. 胶水开发架构（Godot4）

## 8.1 架构原则
- 优先复用成熟插件/服务，不重复造轮子。
- 自研代码仅承担：流程编排、状态协调、输入输出适配。

## 8.2 模块边界
1. **Dialogue Runtime（复用）**
   - 功能：节点驱动、条件跳转、变量读写。
   - 方案：Godot成熟对话系统（如Dialogue Manager类生态）。
2. **State Core（轻自研）**
   - 功能：冲突压力、认知清晰度、信任值等状态容器。
   - 仅维护状态与事件分发，不承载复杂UI逻辑。
3. **AI Adapter（自研胶水）**
   - 功能：自由输入 → 意图标签 → 对话状态机事件。
   - 包含：请求队列、超时回退、敏感词保护、结果缓存。
4. **Save System（复用+少量扩展）**
   - 功能：章节进度、训练成绩、关键选择、设置项。
5. **UI Flow（复用控件）**
   - 功能：对话框、标签卡片、复盘页、设置菜单。

## 8.3 集成策略
- 对话系统为主干，AI Adapter只输出标准化标签，不直接改剧情树。
- 状态变化统一走事件总线（避免UI直接改数据）。
- 关键路径支持离线降级（AI不可用时回到预设选项+规则匹配）。

## 9.2 版本管理
- 主分支：`main`（始终可运行）。
- 开发分支：`dev`。
- 功能分支：`feat/ch01-dialogue-core` 等。
- 对话数据与规则库独立版本号（便于回滚）。

## 9.3 里程碑（第一章）
1. **M1：可移动+可交互原型**（场景漫游+基础对话）
2. **M2：CBT识别闭环**（事实/想法判定+反馈）
3. **M3：AI输入适配**（自由输入接入+安全回退）
4. **M4：训练Boss战可玩**
5. **M5：第一章验收版**（内容、性能、可用性）

## 9.4 内容管线（对话库 + AI交互）
1. 编写训练目标（本段要练什么）。
2. 配置基础对话节点（人工可控主干）。
3. 标注每条节点的认知标签与状态变化。
4. 开放AI输入入口并映射意图标签。
5. QA做“错误输入/极端输入/沉默输入”回归测试。

## 9.5 风险清单
- 内容爆炸：过深分支导致维护失控。
- AI偏题：自由输入返回不稳定。
- 主题风险：心理健康表达不当引发误导。
- 技术耦合：对话、UI、状态互相直接依赖导致难维护。

## 11. 测试与验收标准

## 11.1 功能验收
- 玩家可完成完整第一章流程（开场→训练→Boss→结算）。
- 至少1套完整“事实/想法”教学闭环可重复体验。
- AI输入异常时可降级，不阻断流程。

## 11.2 体验验收
- 首次玩家在引导下能理解“事实 vs 想法”定义。
- 错误反馈不羞辱玩家，复盘可执行。
- UI文本可读、信息层级清晰。

## 11.3 技术验收
- 关键场景稳定帧率（目标值由设备基线定义）。
- 存档/读档一致，不丢失章节状态。
- 对话数据变更不会破坏旧存档读取（版本兼容策略）。
